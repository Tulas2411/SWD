II. Software Requirement Specification 
1. Overall Requirements 
1.1 Context Diagram 
WebMed Healthcare Services System is a web-based platform that provides instant access to 
healthcare services for patients and caregivers. The system integrates core modules such as 
registration/login, health A–Z information, resources (symptom checker, health calculator, doctor 
search, insurance guide, ambulance), online medicine ordering with prescription verification, news & 
experts, payment, and feedback. External entities interacting with WebMed include 
patients/caregivers, doctors, pharmacies, insurance providers, payment gateways, ambulance 
services, and administrators. The context diagram below illustrates the system boundary and the 
main information flows between WebMed and these external actors.

UC - 01: Register 
ID and Name: 
UC-01 - Register 
Created By: 
SangDM / DungTT 
Date Created: 
10/3/2026 
Primary Actor: 
Patient / Caregiver, Doctor 
Secondary Actors: 
Email/SMS Gateway 
Description: 
This use case allows a new, unauthenticated user to create a membership account 
in the WebMed system based on their specific role (Patient, Doctor, or Pharmacist). 
It includes robust validation for age, role uniqueness, password security, and 
mandates an email/phone verification step before granting access. 
Trigger: 
The user clicks the "Register" or "Sign Up" button on the Login interface. 
Preconditions: 
PRE-1: The user does not already possess an active or pending account associated 
with their intended Email or Phone Number. 
PRE-2: The system must be successfully connected to the Email/SMS Gateway for 
identity verification. 
Postconditions: 
POST-1: A new user account is created successfully. 
POST-2: The user receives a confirmation message or email. 
POST-3: The user can log in to the system. 
Normal Flow: 
1.​ The user selects the "Register" option on the login screen. 
2.​ The system prompts the user to select their intended Account Role (e.g., 
Patient/Caregiver, Doctor, Pharmacist). 
3.​ The user selects a role. 
4.​ The system displays the role-specific registration form. Mandatory fields for all 
roles include: Full Name, Date of Birth (DOB), Email Address, Phone Number, 
Password, and Confirm Password. (Note: Doctors/Pharmacists may have 
additional fields like Medical License Number). 
5.​ The user inputs the required data, checks the "I agree to the Terms & 
Conditions and Privacy Policy" checkbox, and clicks "Submit". 
 
ProjectCode – Webmed 
Page 18/147 
 
 

6.​ The system validates the input data against BR-01 (Age), BR-02 (Uniqueness), 
and BR-03 (Password Complexity). 
7.​ The system generates a 6-digit verification OTP and sends it to the user's 
provided Email/Phone via the Email/SMS Gateway. 
8.​ The system displays an OTP Verification screen. 
9.​ The user enters the received OTP and clicks "Verify". 
10.​The system verifies the OTP. 
11.​The system creates the account in the database. 
12.​The system evaluates the chosen role: 
●​ If Patient/Caregiver: The system activates the account, displays a success 
message, and redirects to the Login screen. 
●​ If Doctor/Pharmacist: The system sets the account to "Pending" (BR-07), 
displays a notice that the account requires Admin verification and redirects 
to the Home page. 
Alternative 
Flows: 
9.a User Resends OTP: At step 9, if the user does not receive the OTP, they click 
"Resend OTP". The system invalidates the previous OTP, generates a new one, and 
resends it. The flow resumes at step 9. 
Exceptions Flows: 6.a Underage User (Age Restriction Violation): If the system calculates from the 
DOB that the user is under 18 years old (BR-01), it displays a validation error: "You 
must be at least 18 years old to register a primary account. Minors must be 
registered under a Caregiver's profile." The user must correct the DOB or exit the 
process. 
6.b Account Already Exists: If the entered Email or Phone Number already exists in 
the database (BR-02), the system displays an inline error: "This email/phone number 
is already registered. Please log in or use a different contact." The submit action is 
halted. 
6.c Password Policy Violation: If the entered password fails to meet BR-03, the 
system highlights the missing criteria (e.g., "Password must contain at least one 
uppercase letter and one special character") and halts submission. 
6.d T&C Not Accepted: If the Terms & Conditions checkbox is not checked, the 
system displays an error prompting the user to agree before proceeding. 
 
ProjectCode – Webmed 
Page 19/147 
 
 

10.a Invalid/Expired Verification OTP: If the user enters an incorrect or expired OTP, 
the system displays an error message (“Invalid or expired OTP”) and allows the user 
to retry or request a new OTP (up to system-defined limits). 
Priority: 
High  
Frequency of 
Use: 
High 
Business Rules: 
BR-01, BR-02, BR-03, BR-07 
 
Other 
Information: 
1.​ The user interface must provide a "Toggle Password Visibility" (eye icon) to 
assist users during password creation. 
2.​ Links to the full "Terms and Conditions" and "Privacy Policy" must open in a 
new browser tab/modal to avoid interrupting the registration flow. 
Assumptions: 
The user has stable internet access and immediate access to their email/phone to 
receive the verification OTP.

The Login use-case allows users to authenticate and access the features of the 
WebMed system such as uploading, viewing, and downloading their health-related 
data 
Trigger: 
The user selects the “Login” option or attempts to access a feature that requires 
authentication. 
Preconditions: 
PRE-1: The user already has a registered account in the system. 
PRE-2: The system and authentication service are available. 
Postconditions: 
POST-1: The user is successfully authenticated. 
POST-2: The system grants access to the user dashboard and system features. 
Normal Flow: 
1.​ The user navigates to the Login page. 
2.​ The system displays the login form (Email/Username, Password, "Remember 
Me" checkbox). 
3.​ The user enters their credentials. 
 
ProjectCode – Webmed

4.​ The user submits the login form. 
5.​ The system validates the entered credentials. 
6.​ If the credentials are valid, the system authenticates the user. 
7.​ The system redirects the user to the Dashboard/Home page. 
Alternative 
Flows: 
3a. Forgot Password:  
1. The user clicks "Forgot Password?". 
2. The system redirects to the Password Recovery flow. 
Exceptions Flows: 
5a: Invalid credentials 
1.​ At step 5, if the email/phone or password does not match the database, the 
system displays an inline error message: "Incorrect email or password." 
2.​ The system increments the failed login attempt counter. 
3.​ The user is prompted to try again (Returns to step 3). 
5b: Account locked due to failed attempts 
1.​ At step 5, if the user fails to log in 5 consecutive times, the system

2.​ The system displays a critical error message: "Account temporarily locked 
for 30 minutes due to multiple failed attempts." The user cannot retry until 
the time expires. 
6a: Doctor account pending approval 
1.​ At step 6, if the authenticated user is a Doctor or Pharmacist whose 
credentials have not yet been approved by an Admin (BR-07), the system 
redirects them to a "Pending Approval" holding page instead of the main 
operational dashboard. 
2.​ The system restricts access to core system features until Admin approval is 
granted. 
Priority: 
 High 
Frequency of 
Use: 
 High 
Business Rules: 
BR - 04, BR - 05, BR - 07 
Other 
Information: 
1. Users must have a valid account and remember their credentials.  
2. Passwords must be transmitted and validated using secure hashing algorithms; 
plain-text passwords must never be exposed in the UI or logs. 
Assumptions: 
Users have a stable internet connection to complete the authentication request.

UC - 03: Logout 
ID and Name: 
UC-03 - Log out 
Created By: 
TuanNM / DungTT 
Date Created: 
31/01/2026 
Primary Actor: 
All users 
Secondary Actors: 
None 
Description: 
This use case allows an authenticated user to securely end their active session and 
safely exit the WebMed system, preventing unauthorized access to their account on 
shared devices. 
Trigger: 
The user clicks the "Logout" button on the navigation bar or profile menu. 
Preconditions: 
PRE-1: The user is currently authenticated and has an active session in the WebMed 
system. 
Postconditions: 
POST-1: The user's active session token is permanently invalidated in the system 
backend.  
POST-2: Local storage, session cookies, and cached sensitive data on the user's 
browser/device are cleared.  
POST-3: The user is securely redirected to the Login page. 
Normal Flow: 
1.​ The user selects the "Logout" option. 
2.​ The system intercepts the logout request. 
3.​ The system invalidates the current authentication token and terminates the 
session in the database/server. 
4.​ The system clears associated browser cookies and local cache to prevent 
unauthorized "Back button" navigation. 
5.​ The system redirects the user to the Login interface. 
Alternative 
Flows: 
1a. Automatic Timeout (BR-04): If the system detects 15 minutes of inactivity, it 
automatically triggers step 2. 
Exceptions Flows: 1a: Session already expired: 
 If the user's session has already automatically timed out due to inactivity (BR-04) 
precisely when they click the "Logout" button, the system bypasses the invalidation 
step and directly redirects the user to the Login page with a notice: "Your session 
has already expired. Please log in again." 
2a. Server Connection Error: If the server cannot be reached, the system forces step 
3 (clears local data) and displays: "You have been securely logged out locally. Please 
close your browser."

Priority: 
High 
Frequency of 
Use: 
High 
Business Rules: 
BR-04 
Other 
Information: 
To ensure strict security, after POST-2 is executed, pressing the browser's "Back" 
button must not render any previously viewed secure pages (it should force a 
redirect to the Login page). 
Assumptions: 
None

UC-04 - Recover Password

Description: 
This use case allows an unauthenticated user who has forgotten their password to 
verify their identity via an OTP (One-Time Password) or a reset link sent to their 
registered contact, and subsequently create a new password to regain access to the 
WebMed system. 
Trigger: 
The user clicks the "Forgot Password?" link on the Login interface. 
Preconditions: 
PRE-1: The user must have a previously registered and active account in the 
WebMed system.  
PRE-2: The system must be successfully connected to the Email/SMS Gateway to 
send verification codes. 
Postconditions: 
POST-1: The user's password is successfully updated in the system database.  
POST-2: The user regains access and can log in to the system with the newly created 
password. 
Normal Flow: 
1.​ The user clicks the "Forgot Password?" link on the Login screen. 
2.​ The system displays the Password Recovery screen requesting the user's 
registered email address or phone number. 
3.​ The user enters their registered email/phone and submits the request. 
4.​ The system validates the input format and checks the database for a matching 
account. 
 
ProjectCode – Webmed 
Page 23/147 
 
 

5.​ The system generates a secure, time-bound OTP and sends it to the user via the 
Email/SMS Gateway. 
6.​ The system displays an OTP verification screen. 
7.​ The user enters the received OTP. 
8.​ The system verifies that the OTP is valid and has not expired. 
9.​ The system prompts the user to enter and confirm a new password. 
10.​The user enters the new password twice and submits. 
11.​The system validates the new password against security policies. 
12.​The system updates the password in the database, displays a success message, 
and redirects the user to the Login screen. 
Alternative 
Flows: 
5.a User Requests Reset Link: At step 5, instead of an OTP, the system sends a 
secure, single-use reset token as a URL link via email. The user clicks the link, the 
system validates it, and the flow continues from step 9.  
7.a User Resends OTP: At step 7, the user selects "Resend OTP". The system 
invalidates the old OTP, generates a new one, sends it, and the user continues from 
step 7. 
Exceptions Flows: 4.a Account Not Found: If the entered email/phone does not exist in the system, to 
prevent enumeration attacks, the system displays a generic message "If this contact 
is registered, you will receive a recovery code shortly" and terminates the flow.  
8.a Invalid or Expired OTP: If the OTP is incorrect or expired, the system displays an 
error message. The user is prompted to try again or request a new code.  
10.a Password Policy Violation: If the new password fails to meet the complexity 
requirements, the system displays specific validation errors and prompts the user to 
input a different password. 
Priority: 
High 
Frequency of 
Use: 
Medium 
Business Rules: 
BR-03, BR-05 
Other 
Information: 
A security audit log must record the password recovery event (including timestamp 
and masked User ID). 
Assumptions: 
None

UC - 05: Change Password 
ID and Name: 
UC-05 - Change Password 
Created By: 
DungTT 
Date Created: 
9/3/2026 
Primary Actor: 
Patient / Caregiver 
Secondary Actors: 
None 
Description: 
This use case allows an authenticated user to proactively update their current 
password from within their account settings. 
Trigger: 
The user selects the "Change Password" option in their Account or Security 
settings. 
Preconditions: 
PRE-1: The user is successfully authenticated and currently logged into the WebMed 
system. 
Postconditions: 
POST-1: The user's new password is one-way hashed and securely updated in the 
database.  
POST-2: All other active sessions for this user are forcibly invalidated. 
Normal Flow: 
1.​ The user navigates to the Account Settings and selects "Change Password". 
2.​ The system displays a form requesting the Current Password, New Password, 
and Confirm New Password. 
3.​ The user inputs the data and submits the form. 
4.​ The system verifies that the "Current Password" matches the active password. 
5.​ The system validates that the "New Password" and "Confirm New Password" 
match exactly. 
6.​ The system checks the "New Password" against BR-03 and ensures it is different 
from the current password. 
7.​ The system hashes and updates the new password in the database. 
8.​ The system terminates all other active sessions for this user. 
9.​ The system displays a success message “Password changed successfully!”. 
Alternative 
Flows: 
None 
Exceptions Flows: 4.a Incorrect Current Password: If the user enters the wrong current password, the 
system displays an error message. If the user fails 5 consecutive times, the system 
triggers the security lock and forcibly logs the user out as per BR-05.  
5.a Passwords Do Not Match: The system displays an inline validation error.  
 
ProjectCode – Webmed

Reliability 
The system must be highly reliable, ensuring zero data loss for 
critical transactions such as insurance purchases and doctor

function without failure, as system errors in this module could 
have life-critical consequences.

Usability 
The interface must be intuitive and accessible for all 
demographics, including elderly users or those with limited 
tech skills. To address the "cultural sensitivity" requirement,

easy navigation for the Find a Doctor feature via mobile 
devices.

Performance 
The system must deliver "instant" medical assistance as per the 
project objective. Search results for Doctors and Drugs must 
load in under 2 seconds. The Symptom Checker and Health 
Calculator must provide real-time feedback without latency to 
ensure a smooth user experience. 
5 
Reusability 
Core logic modules, such as the Health Calculator, Symptom 
Checker, and Payment Gateway integration, must be developed 
as independent APIs/Microservices. This ensures they can be 
easily reused for the development of a future Webmed Mobile 
App without rewriting the code. 
6 
Scalability 
The system must be able to auto-scale to handle sudden traffic 
spikes, such as during a health crisis (when traffic to News and 
Experts increases) or during insurance renewal seasons, 
ensuring the server does not crash under high concurrency.

99.9% uptime (24/7 availability). Critical functions like
